3. Scaling & Encoding (Don't Skip)

The distance metric you choose dictates how you must prepare your features.

Numeric Features

visc_hot, vol_ml

→ Standardize (Z-score) or Range-Normalize.

Categorical Features

api_X, jaso, base_group, country

→ One-Hot Encode (for Euclidean) OR Leave as Categories (for Gower).

Binary Flags

..._present

→ Already 0/1; acts as a matching coefficient.

4. Distance Choices That Fit This Dataset

Option A: One-Hot + Euclidean

  • Build a design matrix: Scale numeric + one-hot encode all categories.
  • Pro: Required for algorithms like K-Means.
  • Con: Can create high-dimensional sparsity (many columns) with few rows (only 7).
  • Use with: K-Means (Euclidean), K-Medians (Manhattan).

Option B: Mixed-Type / Gower

  • Keep types separate: Range-normalize numeric; use binary matching for categories.
  • Pro: Handles mixed data types natively without one-hot encoding.
  • Con: Less common in basic ML packages.
  • Use with: DBSCAN, Hierarchical Clustering (e.g., single-linkage).

5. Sanity Checks Before You Compute

Remove Constants

Always remove constant columns (visc_w, container) from distance calculations. They add no information and waste computation.

Check Imbalance

Be aware of highly imbalanced features (e.g., 6-1 or 5-2 splits). Don't let them accidentally over-weight your distance metric.

Verify Spelling

The data contains exact spellings like Thiland. Do not "fix" this silently. If you normalize, document it explicitly.
