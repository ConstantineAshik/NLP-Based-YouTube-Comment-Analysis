From Visuals to Vectors

Analyzing and Comparing Music Video Clips

The Goal: Classification

Our primary objective is to build a system that can automatically classify and compare short music video clips based on their visual content. We aim to distinguish between a "solo home practice" and a "staged band performance."

🎯 Target & ⛓️ Constraints

  • Distinguish Clip Types: Create a system that can tell the difference between distinct scenes, like a bedroom practice vs. a live show.
  • Handle New Content: The system must be extensible to analyze unseen clips, such as one featuring only a vocalist.
  • Overcome Variability: The model must be robust against challenges like variable lighting, occlusions, and diverse environments.

🔑 Guiding Principle

Stable Schema, Flexible Values. Our data structure must be consistent, but able to represent missing or absent information cleanly without inventing data.

Core Attributes: A Robust Foundation

These are task-agnostic features that provide a descriptive baseline for any video clip. This structure is designed to be comprehensive yet flexible.

clip_id (string): Unique identifier.
people_count (integer): Number of visible people.
instrument_types (set): Visible instruments.
scene (enum): Environment type (home, stage).
camera_angle (enum): Camera perspective.
brightness (float): Average pixel intensity.
motion_magnitude (float): Frame-to-frame movement.
quality_flag (enum): Data quality notes (e.g., occluded).

Extensibility: Planning for the Future

Hard-coding every possible feature is brittle. We use linked tables for detailed or experimental data to avoid "schema churn" — constantly redesigning our main data table.

Smart Encoding: From Sets to Vectors

For a model to understand categorical data like instrument types, we must convert it into a numerical format. Multi-hot encoding is a powerful choice for this.

Set Form

(Human-Readable)

L001: {guitar}

R001: {guitar, drums}

Multi-Hot Vector

(Model-Friendly)

L001: [0, 1, 0]

R001: [1, 1, 0]

Vocabulary: [drums, guitar, keyboard]
Why is this better? A simple count (e.g., `num_instruments = 2`) loses **which** instruments are present. A vector preserves this detail, allowing models to learn instrument-specific patterns.
