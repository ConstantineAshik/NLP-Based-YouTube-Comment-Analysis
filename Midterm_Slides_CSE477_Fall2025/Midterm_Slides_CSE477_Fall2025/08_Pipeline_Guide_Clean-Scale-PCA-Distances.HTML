Steps 1-3: Data Definition & Cleaning

The Teach-Ready Pipeline

1. Define the Schema

Intent: Consistent & Extensible

  • Create a schema that avoids hard-coding every possibility.
  • Avoid "schema churn" by keeping the structure consistent yet extensible.
  • Allow **nullable fields** where appropriate to represent missing data cleanly.
  • Include a dedicated "data quality notes" field.
  • **Discipline:** Do not invent values for fields that don't fit.

Fallback: If a clip won't fit, store essential attributes and use the "notes" field to describe the discrepancy.

2. Extract Features

Intent: Faithful Signal

  • Turn raw media (clips) into analyzable, structured attributes.
  • Check for unreliable signals (e.g., occlusion, low-light) or detector errors.
  • If the signal is unreliable, **set fields to missing**, not fabricated or guessed values.

Fallback: If extraction is unreliable, mark data as missing and defer the problem to the cleaning step.

3. Clean & Repair

Intent: Trustworthiness

  • Follow the "Full Data Repair Workflow."
  • Inspect data for impossible values (e.g., `people_count=0` when people are visible).
  • Correct obvious, simple errors.
  • If an error is uncorrectable or uncertain, set the value to **missing**.
  • Document all repairs and missing-value decisions.

Fallback: If uncertain about a value, always prefer marking it as missing over guessing.

Steps 4-5: Transformation

4. Normalize / Standardize

Intent: Fair Comparison

  • Min-Max Scaling: Use for simple, bounded ranges. (Warning: sensitive to outliers).
  • Z-score Scaling: Prefer when data are larger or outlier-prone.
  • **No Leakage:** Fit parameters (mean, std, min, max) on the **training set ONLY**.
  • Reuse those same parameters to transform the validation and test sets.

Fallback: If unsure, choose Z-score (as it's more robust) and document the choice.

Caption: An outlier (Bin 5) drastically changes the 'max' value, distorting Min-Max scaling for all other data. Z-score is more robust to this.

5. Reduce Dimensionality

Intent: Simplicity without losing signal

Use Feature Selection when:

  • **Interpretability is a priority.**
  • You need to remove redundant features.

Use PCA when:

  • Performance is the priority.
  • You have many correlated numeric features.
  • Must be applied *after* scaling.
  • Must evaluate impact (unsupervised).
  • Fit on **train-only** to avoid leakage.

Fallback: Combine methods. Run Feature Selection first, then apply PCA to the remaining features.

Steps 6-7: Similarity & Final Checks

6. Choose Similarity / Distance

Intent: Match metric to data type

Select the correct metric based on the nature of the features:

Numeric Features:
Euclidean (must be applied *after* scaling).
Text / Metadata:
Cosine (for conceptual similarity, measures angle not magnitude).
Categorical Labels:
Hamming (only if labels are clean and consistent; brittle with noise).
Sets / Binary:
Jaccard (measures intersection over union).

Fallback: For mixed data, calculate per-type distances and combine them using a **weighted sum**.

7. Guardrails & Final Check

Intent: Ensure Defensible Results

Before proceeding, perform a final sanity check:

  • Re-verify NO DATA LEAKAGE. Confirm all transforms (scaling, PCA) were fit on the **training set only**.
  • Document all trade-offs chosen (e.g., interpretability vs. performance).
  • Ensure data quality notes are populated.

Key Trade-offs

Interpretability vs. Performance

Feature Selection preserves original features, while PCA mixes them into components.

Angle vs. Magnitude

Cosine measures conceptual similarity (direction), while Euclidean measures "as-the-crow-flies" distance.

Label Quality

Hamming distance is fast but requires pristine, consistent labels to be meaningful.

Item Importance

Jaccard treats all items in a set with uniform, equal weight.

Common Pitfalls

When It Breaks: Common Pitfalls

Identify potential failures and their prescribed solutions.

Problem: Schema Churn

Hard-coded features cause constant redesign when new data types appear.

Fix: Use an extensible schema with nullable fields and "missing" handling.

Problem: Outliers Wreck Min-Max

A single large outlier compresses the entire valid range of data.

Fix: Prefer Z-score scaling, which is more robust to outliers.

Problem: PCA Harms Interpretability

PCA components mix original features, making them impossible to explain.

Fix: Use Feature Selection first if interpretability is a priority.

Problem: Unscaled Euclidean Distance

Distance is dominated by the feature with the largest scale (e.g., 'salary' vs. 'age').

Fix: Always normalize/standardize numeric features before using Euclidean.

Problem: Hamming on Noisy Labels

An "all-or-nothing" metric that heavily punishes a single typo or inconsistency.

Fix: Clean labels first, or switch to a more robust "soft" metric.

Problem: Jaccard Ignores Importance

Treats all items in a set as equally important (e.g., 'rare-tag' == 'common-tag').

Fix: Engineer critical items as separate features if weighting is needed.

Key Takeaways

Discipline to Follow (Memorize)

The core principles for a defensible data pipeline.

Never invent values; always mark as missing. (Safe Passage: Missing > Guessing)

Fit scalers/PCA on training data ONLY. Reuse those parameters on validation and test sets to prevent leakage.

Normalize numeric data before applying Euclidean distance or PCA.

Prefer Z-score if outliers are present or expected; document the parameters.

Choose Selection for interpretability; choose PCA for correlated numerics (or combine them).

The metric must match the data type (Euclidean, Cosine, Hamming, Jaccard).

Record data-quality notes and all design decisions.

Avoid hard-coding features; keep the schema extensible.
